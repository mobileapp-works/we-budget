## 🔴 重大（お金・データ・主要フローが壊れる）

### R-1. パスワード再設定が最後まで完了できない（フロー断絶）
- [supabaseBackend.ts:122](../src/data/supabaseBackend.ts#L122) は `redirectTo: 'webudget://reset-password'` でメールを送るが、アプリ側に**リンクを受けてセッション化する処理が一切ない**（`createSessionFromUrl` / `exchangeCodeForSession` / `Linking` ハンドラともに不存在）。さらに「新しいパスワードを入力する画面」も存在しない（[reset-password.tsx](../app/(auth)/reset-password.tsx) は送信フォームのみ。設計 2c の「新パスワード設定」が未実装）。
- 結果: メールのリンクを踏んでもアプリが開くだけで何も起きない。**パスワードを忘れたユーザーは永久に復帰不能**。プロフィールの「パスワードを変更」も同じリセットメール方式なので同様に完結しない。
- 対応案: recovery ディープリンクの受信（URLからトークン交換）→「新パスワード入力」画面 → `auth.updateUser({ password })` の3点セット。

### R-2. 共同口座の残高が翌月から狂う（お金バグ）
- [shared-account.tsx:21](../app/shared-account.tsx#L21) が共同口座払いの支出を `useExpenses(getMonthKey())`（**当月分のみ**）で取得して残高計算に渡している。入金・出金は全期間なのに支出だけ当月分 → **月が変わると過去の共同口座支出が残高から消え、残高が実際より多く表示される**。
- 設計（design.md「共同口座残高の計算」）は全期間のΣ。専用の全期間集計（またはサーバー側集計）が必要。

### R-3. 為替レートを入力するUIが存在しない（要件7-9未達）
- `useExchangeRateActions` / `Backend.upsertExchangeRate` / DBテーブル・RLSまで完備しているのに、**どの画面からも呼ばれていない**（`app/` 内で使用ゼロ）。
- 通貨チップは8通貨（[constants/index.ts:22](../src/constants/index.ts#L22)）選べるため、外貨支出を作った瞬間に「レート未設定で精算に含まれていません」警告が出続け、**設定する手段がないので外貨支出は精算・レポート・予算・共同残高から永久に除外**される。
- レポート/予算/ホーム集計では警告すら出ず**無言でスキップ**（[report.tsx:45](../app/(tabs)/report.tsx#L45)、[budget.tsx:42](../app/budget.tsx#L42)）。合計が過少表示になる。

### R-4. user2側で負担割合の表示・保存が反転する（お金バグ）
- [profile.tsx:120-171](../app/profile.tsx#L120-L171) は `splitRatioUser1` を常に「自分 : パートナー」として表示し、選択値をそのまま `update_split_ratio(p_user1_percent)` に渡す。
- **自分が user2（招待コードで参加した側）の場合、「60 : 40」を選ぶと実際は パートナー=60% / 自分=40% になり意図と逆**。表示も同様に逆転。精算金額が逆配分で計算される。
- 対応案: `session.userId === pair.user1Id` を見て変換してから表示・送信する。

### R-5. レシート画像がStorageにアップロードされない（要件#3・7-8未達）
- [expense-input.tsx:153](../app/expense-input.tsx#L153) は撮影画像のローカルURI（`file://...`）をそのまま `receipt_image_url` としてDBに保存。**レシート用のアップロードAPIが Backend インターフェースごと存在しない**（avatar / categoryIcon は実装済みなのにレシートだけ欠落）。
- `receipts` バケットとRLS（0003）は準備済みで宙に浮いている。
- 結果: パートナーの端末ではレシート画像が表示されない（壊れたImage）。本人も再インストールで消える。「後から閲覧可能」の要件を満たさない。

### R-6. ペア移行後、別pairの支出が一覧・集計に混入する
- [supabaseBackend.ts:252-265](../src/data/supabaseBackend.ts#L252-L265) `listExpenses` に **`.eq('pair_id', pairId)` がない**（設計のAPI仕様には明記あり）。
- expenses の RLS は `pair_id = get_my_pair_id() OR recorded_by = auth.uid()`（解除後閲覧の要件7-10のため）。この OR により、**ペア参加/解除を経験したユーザーは「旧pair・ソロ時代に自分が記録した支出」が新しい家計簿の履歴・ホーム合計・レポート・予算使用額に混入**する。
- 一方 精算RPCは pair_id で絞るため、**ホームの合計と精算残高で数字が食い違う**。設計の「参加者のソロ時代の支出は新pairに持ち込まない」に反する。
- 対応案: `listExpenses` に pair_id フィルタを追加し、「過去の記録」は別導線（設計の 7-10 閲覧用）として扱う。

### R-7. 精算実行で「支出が編集されました」通知がN連発 + 肝心の精算通知が未実装
- `execute_settlement`（0004）の `update expenses set settlement_id=...` が [0005 の on_expense_change](../supabase/migrations/0005_fixed_cost_cron.sql#L63) の UPDATE 分岐を**スタンプ対象の行数ぶん発火** → パートナーに `expense_edited` 通知（＋プッシュ）が大量に届く。
- しかも **settlements INSERT に対する通知トリガーがどこにもない** = 要件7-6「精算実行→両者にプッシュ通知（type=settlement）」が未実装。通知設定に「精算の通知」トグルだけが存在する。
- 対応案: on_expense_change で「settlement_id のみの変更」をスキップ + settlements INSERT トリガーで両者に settlement 通知。

### R-8. 予算アラート（80% / 100%）が完全未実装（要件#9・7-3未達）
- 設計の DBトリガー `check_budget_threshold` がどのmigrationにも存在しない。クライアント側もバー表示のみで通知生成なし。
- **MVP優先度「高」の機能が丸ごと欠落**。通知設定画面には「予算アラート」トグルがあるのに一度も発火しない（ユーザーを欺く見た目になっている）。

### R-9. 月末精算リマインドが未実装（要件7-6・#5未達）
- `settlement_reminder` を生成する cron がない（0005 は固定費計上と変動リマインドの2本のみ）。これも通知設定にトグルだけ存在する。

### H-5. 固定費フォームの機能不足（要件7-2に対して）

### H-6. 予算を削除・解除できない

### H-9. join_pair がペア済みユーザーの参加を弾かない（データ整合性）
