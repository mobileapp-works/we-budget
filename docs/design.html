ソロモード（ペアリング前）

サインアップ時に本人のみの pair を自動生成。立替精算・共同口座・パートナー通知・負担割合は非表示。支出記録・予算・レポート・固定費は1人でも利用可能。設定の「パートナーを招待」からいつでもペアリング可。

テーブル: shared_account（共同口座）

カラム名型説明制約
id / pair_iduuidID / ペアPK / FK CASCADE
typetext種別CHECK: 'deposit','withdrawal'
user_iduuid入金者/記録者FK SET NULL
amount / currencynumeric / text金額 / 通貨CHECK: > 0

役割: 入金(deposit)と支出計上しない現金移動(withdrawal: ATM引き出し等)専用。共同口座での買い物は expenses(is_shared_payment=true)に一本化（二重入力回避）。
残高 = Σ(deposit) − Σ(withdrawal) − Σ(expenses WHERE is_shared_payment=true)（JPY換算）

テーブル: budgets（予算）

カラム名型説明制約

UNIQUE(pair_id, category_id)（全体予算は category_id IS NULL の部分一意で1件）。多通貨は exchange_rates で予算通貨に換算して集計。

テーブル: exchange_rates（為替レート）

カラム名型説明制約
id / pair_iduuidID / ペアPK / FK CASCADE
from_currency / to_currencytext元通貨 / 先通貨NOT NULL / default 'JPY'

UNIQUE(pair_id, from_currency, to_currency)。MVPは最新の1件を使う簡易方式（支出日時点のレート履歴は持たない）。

テーブル: notifications / notification_settings

notifications: user_id, pair_id, type（expense_added / expense_edited / expense_deleted / settlement / reminder_variable / budget_warning / budget_exceeded / settlement_reminder）, title, body, data(jsonb), is_read。
notification_settings: ユーザーごと（UNIQUE(user_id)）に上記種別のON/OFFを保持。全 default: true。

RLS（Row Level Security）方針

テーブルSELECTINSERTUPDATEDELETE
profiles自分 + パートナー自分のみ自分のみ自分のみ
pairs自分のペア認証ユーザー自分のペア自分のペア
categories自分のペア自分のペア自分のペア自分のペア（is_default不可）
expenses自分のペア または recorded_by=自分（解除済pairの自分の記録も可）自分のペア自分のペア かつ未解除自分のペア
settlements自分のペア自分のペア——
notifications自分宛のみサーバーのみ自分宛（既読）—
RLS再帰回避: 「同じペア」判定が profiles/pairs を直接サブクエリすると無限再帰。get_my_pair_id() SECURITY DEFINER ヘルパーで pair_id を取得し pair_id = get_my_pair_id() で判定する。

データ整合性・退会/解除の方針

項目方針
楽観的ロックexpenses / fixed_costs / shared_account は updated_at で衝突検知
トランザクション精算実行は RPC で原子的に実行
削除方針論理削除（deleted_at）
ON DELETEpair配下=CASCADE / ユーザー参照=SET NULL（匿名化）/ category参照=RESTRICT
退会時共有データは pair の資産として保持。ユーザー参照は SET NULL → UIで「退会したユーザー」表示
ペア解除時元pairは deleted_at で読み取り専用化。各自に新ソロpair発行。recorded_by=自分 の記録は閲覧可（要件7-10）

立替残高ロジック（RPC）

calculate_settlement_balance(pair_id): 1. 未精算かつ個人払いの支出を取得 WHERE settlement_id IS NULL AND is_shared_payment=false AND deleted_at IS NULL 2. 外貨は exchange_rates でJPY換算（未設定は警告） 3. user1_paid / user2_paid を集計 4. 本来負担 = total × split_ratio / 100 5. 精算額 = ABS(user1_paid − user1_should_pay) 払う側/受取側を決定 返り値: { settlement_amount, from_user_id, to_user_id, currency:'JPY', unconverted_currencies:[] } execute_settlement(pair_id): ※トランザクション 1. calculate で金額を再計算（クライアント値を信用しない） 2. settlements に1件INSERT 3. 対象 expenses に settlement_id をスタンプ → 残高リセット

ペアリング時のデータ統合方針

MVPは「招待した側(user1)のpairに参加側(user2)が合流」。参加者のソロ時代の支出は本人のソロpairに残し新pairには持ち込まない（取り違え回避）。オンボーディングで「招待した人の家計簿に合流します」と明示。

🏗️ Design Document

📱 WE BUDGET 📅 Created: 2026-06-22 🔄 Updated: 2026-06-22 👤 taishirou16

1. Screens (18)

Splash, Login, Sign Up, Verify Email, Reset Password, AI Consent, ATT/UMP Consent, Pairing, Home, Expense Input, Expense List, Expense Detail, Settlement, Shared Account, Fixed Costs, Budget, Report, Notifications, Categories, Profile, Settings.

2. Navigation

Main: Expo Router tabs (Home / History / Input / Report / Settings). Auth: stack. Splash restores session → Home or Login. Solo mode available before pairing (a solo pair is auto-created on signup).

3. Data Model (11 tables)

profiles, pairs, categories, expenses, fixed_costs, settlements, shared_account, budgets, exchange_rates, notifications, notification_settings. Soft delete (deleted_at), optimistic locking (updated_at), RLS per pair via get_my_pair_id() SECURITY DEFINER to avoid recursion. ON DELETE: pair data CASCADE, user refs SET NULL (anonymize), category refs RESTRICT. See design.md for full schema.

4. API / Logic

Auth: Email + Apple + Google (session restore, email verification, forgot-password, Apple name fallback). RPCs: handle_new_user, join_pair, leave_pair, calculate_settlement_balance, execute_settlement. Settlement uses settlement_id stamping (no time cutoff) so backdated entries are never lost. Shared-account purchases are recorded once as expenses(is_shared_payment=true). Server-side notification pipeline gated by notification_settings; budget 80%/100% via check_budget_threshold trigger.

5. State

Zustand (user/profile/pair/theme/language), TanStack Query (all server data with staleTime), useState (forms/loading/modals/OCR).

6–11. Common Standards

i18n (ja/en), Responsive (iPhone SE–iPad), 4 Screen States, Accessibility (WCAG AA, 44pt), Sentry monitoring, FlatList performance, App Store compliance (privacy manifests, ATT). Online only.
