## リリース・ビルド

### EAS ビルドの所要時間は大きくブレる（5分〜2日超）。当てにせず先に投げる
- **発見文脈**: WeBudget v1.0.0 の提出準備（2026-07-14）。
- **問題/知見**: 同じ production プロファイルでも **build#16/#17 は 2日以上**（7/11開始→7/13完了）かかったのに、**build#21 は5分**で完了した。所要時間はキュー混雑次第で桁違いに変動する。
  - ⚠️ **教訓（メタ）**: 過去2件だけを見て「EASは2日かかる」と一般化しかけたが誤りだった。**n=2の実績から所要時間を法則化しない**。工程見積もりに使うなら実測を都度取る。
- **対策/パターン**: 所要時間が読めない以上、**ビルドは工程の最初に `--no-wait` で投げておく**のが常に有利（早ければ即次工程、遅くても待ち時間に掲載文・審査メモ・スクショ計画を書ける）。ビルド完了が前提の作業（スクショ撮影・Privacy Manifest の集約確認・TestFlight実機確認）は後段にまとめる。
- **反映先**: `09_release/SKILL.md`（手順の順序を「ビルド先行」に。所要時間は明記しない）

### 広告ユニットIDは「本番ID未設定ならテストIDにフォールバック」が事故らない
- **発見文脈**: 提出前の広告ID確認。
- **問題/知見**: `src/lib/ads.ts` が `__DEV__ || IS_MOCK` ならテストID、本番ビルドでは env の本番ID、**未設定ならテストIDへフォールバック**する設計になっていた。これにより「本番IDで自己表示してBAN」も「未設定でクラッシュ」も防げる。本番IDは `eas.json` の production プロファイルの env にのみ置く。
- **対策/パターン**: 全アプリでこの3段構え（dev=テスト / prod=env本番 / 未設定=テストにフォールバック）をデフォルトにする。
- **反映先**: `07_monetization/SKILL.md` または `GLOBAL_STANDARDS §10`

### App Privacy は「同梱SDKの .xcprivacy を実読して転記」が唯一の正解
- **発見文脈**: WeBudget v1.0 の App Privacy 入力（2026-07-15）。
- **問題/知見**: ASC の申告が SDK の自己申告とズレると 5.1 系リジェクト要因。推測で埋めると必ずズレる。
  - **IPAを解凍して `Payload/<App>.app/<SDK>.bundle/PrivacyInfo.xcprivacy` を読めば、SDKが何を申告しているか正確に分かる**（AdMob なら `GoogleMobileAdsResources.bundle`）。
  - 実際 AdMob は `DeviceID / AdvertisingData / ProductInteraction / CoarseLocation / CrashData / PerformanceData / OtherDiagnosticData` の7種を申告していた。**自社コードで位置情報もクラッシュ収集もしていなくても、同梱SDK分は申告義務がある**。
  - **`Tracking = true` は `DeviceID` だけ**で `AdvertisingData` は false だった。「広告系は全部トラッキング」と思い込むと誤る。
- **対策/パターン**: ビルド後に IPA を解凍 → 各 SDK の `.xcprivacy` を読む → その内容を ASC にそのまま転記する。手順:
  ```
  # IPA は zip。Payload/<App>.app/ 配下の *.xcprivacy を全部拾って読む
  ```
- **反映先**: `09_release/SKILL.md`（App Privacy の章に「IPAから転記する」手順を追加）

### 家計簿・金融系アプリは App Privacy の「その他の金融情報」を申告する
- **発見文脈**: 同上。支出データを「その他のユーザーコンテンツ」だけで申告しかけた。
- **問題/知見**: Apple の「Other Financial Info」の定義は「salary, income, assets, debts, **or any other financial information**」。**家計簿の支出額・予算・精算額・口座残高は該当する**。ユーザーコンテンツ扱いだけで済ませると過少申告になる。
- **対策/パターン**: 金額を扱うアプリは `金融情報 > その他の金融情報` を必ず申告（用途=アプリの機能 / リンク=はい / トラッキング=いいえ）。**過少申告は 5.1.1 でリジェクトされるが、過剰申告では落ちない**ので迷ったら申告する。
- **反映先**: `09_release/SKILL.md` / `GLOBAL_STANDARDS §5`

### スクショの要求サイズは ASC の実画面で確認する（「6.9インチ必須」を鵜呑みにしない）
- **発見文脈**: WeBudget v1.0 提出（2026-07-15）。28枚を 1290×2796（6.9"）で作って ASC にアップしようとしたら「寸法が正しくありません」で弾かれた。
- **問題/知見**: `09_release/SKILL.md` は「**6.9インチ iPhone（1290×2796）: 必須**」「13インチ iPad（2064×2752）」と断定していたが、**実際の ASC は iPhone 6.5インチ（1242×2688 / 1284×2778）スロットしか出さなかった**。要求サイズはアプリ・時期・ASCの状態で変わる。
- **対策/パターン**:
  1. **スクショを作る前に ASC の「プレビューとスクリーンショット」を開いて、実際に表示される寸法をメモする**。ドキュメントの数字を信用しない。
  2. 出力サイズは**スクリプトの定数1箇所**にまとめておく（`scripts/make-store-screenshots.js` の `DEVICES`）。サイズ変更が1行＋再実行で済み、28枚の作り直しが30秒で終わった。手作業やAI画像編集だと全滅していた。
- **反映先**: `09_release/SKILL.md`（サイズを断定している記述を「ASCで実測してから作る」に修正）/ `08_aso/SKILL.md`

### ストア掲載の言語ごとにスクショ欄が別（ASCの言語セレクタを見落とすと混ざる）
- **発見文脈**: 同上。ASC のスクショ画面は右上の言語セレクタ（例「英語（アメリカ）」）で編集対象のローカライズが変わる。
- **問題/知見**: セレクタを見ずにアップすると、英語ページに日本語スクショが載る。日英対応アプリでは **スクショは言語×デバイスの掛け算**（今回 2言語×2デバイス×7枚＝28枚）。
- **対策/パターン**: 出力を `out/<lang>-<device>/` に分けておき、ASC側の言語セレクタと1対1で対応させる。
- **反映先**: `08_aso/SKILL.md`

### 英語スクショはDBのデータが日本語のまま残る（UI翻訳では消えない）
- **発見文脈**: 英語版スクショ撮影時。
- **問題/知見**: `t()` で翻訳されるのは UI 文字列とデフォルトカテゴリ名・支払者ラベルのみ。**DBに保存された値（パートナー表示名・店名・固定費名・メモ）は英語UIでも日本語のまま**出る（`ExpenseRow` は `expense.storeName || categoryName`）。結果「ユーザー owes you」のような混在が起きる。
- **対策/パターン**: 英語スクショ用のデモデータは、
  - **店名を空にする**とカテゴリ名（翻訳される）が出るので両言語で自然になる ← 効果大
  - パートナー表示名は英語圏でも通じる名前にする（"Kana" 等）。デフォルト名のままだと「ユーザー」が露出する
- **反映先**: `08_aso/SKILL.md`（スクショ用デモデータの作り方）/ `GLOBAL_STANDARDS`（i18n方針）

### スクショはApple 2.3.3があるのでAIに「UIを描き直させない」
- **発見文脈**: スクショをChatGPTで装飾する運用を設計（`docs/screenshot_plan.md`）。
- **問題/知見**: 生成AIに画像編集させると**UI内の文字・数字を勝手に作り替える**ことがある。スクショが実画面と食い違うと **Guideline 2.3.3** でリジェクト対象。
- **対策/パターン**: プロンプトに「アップロードした実スクショは改変・再描画しない。周囲の背景と見出しのみ追加」「書き出しは正確に1290×2796」を明記する。歪む場合は Canva/Figma で実スクショを素材として合成する方が確実。
- **反映先**: `08_aso/SKILL.md`（スクショ戦略に「AI編集時の2.3.3対策」を追記）
